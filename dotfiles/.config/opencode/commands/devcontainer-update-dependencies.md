# Update Devcontainer Dependencies

**Arguments:** "$ARGUMENTS"

## Behavior

- With no arguments, inspect both known devcontainers:
  1. Reserhub Revenue Full: `/Users/ignacio/repositories/reserhub-revenue-full`
  2. Adara CRM: `/Users/ignacio/repositories/adara-crm`
- With a project argument, inspect only that project.
- Accepted project keys: `reserhub`, `reserhub-revenue`, `reserhub-revenue-full`, `adara`, `adara-crm`.
- Before editing any tool version, produce a planned-upgrades report and ask the user to accept it.
- Do not change dependency, package, feature, plugin, or tool-version pins until the user explicitly approves the planned upgrades.
- Keep both devcontainers aligned unless a project-specific constraint requires a different approved version.

## Scope

Review pinned versions in these locations for each selected project:

- `.devcontainer/Dockerfile` and any Dockerfile copied or included by the devcontainer.
- `.devcontainer/setup.sh` and related setup scripts.
- `.devcontainer/devcontainer.json` features and feature options.
- `.devcontainer/docker-compose.devcontainer.yml` images, build args, and versioned tooling inputs.
- Tooling policy files that document or constrain devcontainer tool versions.
- OpenCode plugin pins in the host OpenCode config.
- Local host OpenCode plugin package installs, including `~/.config/opencode/package.json` when present.
- `.devcontainer-persistent/config/opencode/package.json` when selected devcontainers install matching OpenCode plugin packages.

## Planned-upgrades report

For every current pin, report a table with:

| Project | File | Tool or package | Current pin | Latest target | Source checked | Notes |
|---|---|---|---|---|---|---|

Use package registries, release APIs, official docs, or repository releases as the source for latest versions. If latest cannot be verified, mark the item as blocked instead of guessing.

After the report, ask one approval question with these choices:

1. Apply all planned upgrades.
2. Apply only selected upgrades.
3. Do not apply changes.

Stop at this gate until the user answers.

## Apply workflow

After approval:

1. Apply only the approved version changes.
2. Update the matching tooling policy text when a tool version or update rule changes.
3. If an OpenCode plugin pin changes, update all matching local plugin declarations:
   - `dotfiles/.config/opencode/opencode.json`
   - `~/.config/opencode/package.json` when it declares the plugin package.
   - `.devcontainer-persistent/config/opencode/package.json` for each selected devcontainer that installs the plugin package.
4. Keep lockfile changes only when generated by the selected package manager for an approved package change.
5. Run changed-scope formatting, linting, type checking, and related tests for each edited repository according to that repository's instructions.
6. Rebuild or run the affected devcontainer only when needed to verify the approved dependency changes.
7. Summarize applied upgrades, skipped or blocked pins, verification results, and any rebuild command the user should run next.

## Examples

```text
/devcontainer-update-dependencies
/devcontainer-update-dependencies reserhub
/devcontainer-update-dependencies adara
```
